Gunicorn 启动服务

约 3148 字大约 10 分钟

2025-01-21

简介

Gunicorn (Green Unicorn) 是一个 UNIX 下的 WSGI HTTP 服务器，它是一个 移植自 Ruby 的 Unicorn 项目的 pre-fork worker 模型。它既支持 eventlet, 也支持 greenlet。WSGI(Web Server Gateway Interface) 即：Web服务器网关接口。

安装

$ pip3 install gunicorn

运行

$ gunicorn -w 4 -b 127.0.0.1:8000 -k uvicorn.workers.UvicornWorker main:server

  • -w 4：指定工作进程的数量。通常，这个数字应该设置为可用CPU核心的2-4倍。调整这个数字可以优化应用的性能。
  • -b 127.0.0.1:8000 服务启动在本地的8000端口
  • -k uvicorn.workers.UvicornWorker：指定使用Uvicorn的工作类，这使得Gunicorn能够异步地运行ASGI应用。
  • main:server：指定应用的模块和应用实例名称。main是文件名（不包括.py扩展名），server是你在main.py中创建的FastAPI实例的变量名

gunicorn有四种方式开启服务

$ gunicorn -w 4 demo:app --worker-class sync
$ gunicorn -w 4 demo:app --worker-class gevent
$ gunicorn -w 4 demo:app --worker-class tornado
$ gunicorn -w 4 demo:app --worker-class eventlet

gunicorn的工作模式一般分为同步worker使用和异步worker使用。

使用配置文件运行

'''
Author: matiastang
Date: 2024-04-09 14:45:10
LastEditors: matiastang
LastEditTime: 2024-04-23 17:08:16
FilePath: /kk-aippt/gunicorn_config.py
Description: gunicorn配置文件
'''
import multiprocessing

# 并行工作进程数, int，cpu数量*2+1 推荐进程数
workers = multiprocessing.cpu_count() * 2 + 1
# 绑定的ip与端口
bind = "127.0.0.1:8000"
# 工作模式，sync (默认值) eventlet gevent tornado（这个是指定worker为Uvicorn的Worker，为fastapi专属，其他比如flask应用不需要带上这个参数。）
worker_class = "uvicorn.workers.UvicornWorker"

这里使用的是 cpu数量*2+1 个工作线程。

同步worker（Sync Worker）

默认的、最简单的worker模式，就是同步模式。 每个worker进程，一次只处理一个请求。如果此时又有其他请求被分配到了这个worker进程中，那只能被堵塞了，只能等待第一个请求完成。并且一个请求一个进程，并发时，是非常消耗CPU和内存的。

注意 因此，只能适合在访问量不大、CPU密集而非I/O的情形。但是也有好处，好处就是，即使一个worker的进程crash了，也只会影响到一个请求。不会影响其他的请求

异步workers（Async Worker）

异步worker有Gevent和Eventlet两种，都是基于Greenlet实现的。

当使用了异步worker，就能同时处理不止一个请求，就不会出现上面同步worker那样，当一个请求就把后续请求都block堵塞住了。

提示 gunicorn允许通过设置对应的worker类来使用这些异步Python库

如下，在单核机器上运行的 gevent

$ gunicorn --worker-class=gevent --worker-connections=1000 --workers=3 main:app
  • worker-connection 是对于 gevent worker 类的特殊设置，每个workers的最大连接数。
  • --workers 工作线程数，建议为（2CPU）+1。因为这里是单核，我们设置的是3个worker。

在上面这种情况下，最大的并发请求数是3000（worker数3*每个worker的最大连接数1000）。

并发方式

为了提高使用 Gunicorn 时的性能，我们可以使用下面 3 种并发方式。

第一种并发方式（workers 模式，又名 UNIX 进程模式）

每个 worker 都是一个加载 Python 应用程序的 UNIX 进程。worker 之间没有共享内存。建议的 workers 数量是 (2*CPU)+1。

对于一个双核（两个CPU）机器，5 就是建议的 worker 数量。

$ gunicorn --workers=5 main:app

第二种并发方式（多线程）

Gunicorn 还允许每个 worker 拥有多个线程。在这种场景下，Python 应用程序每个 worker 都会加载一次，同一个 worker 生成的每个线程共享相同的内存空间。为了在 Gunicorn 中使用多线程。我们使用了 threads 模式。每一次我们使用 threads 模式，worker 的类就会是 gthread：

$ gunicorn --workers=5 --threads=2 main:app

上一条命令等同于：

$ gunicorn --workers=5 --threads=2 --worker-class=gthread main:app

在我们的例子里面最大的并发请求数就是 worker * 线程，也就是10。

在使用 worker 和多线程模式时建议的最大并发数量仍然是(2*CPU)+1。 因此如果我们使用四核（4 个 CPU）机器并且我们想使用 workers 和多线程模式，我们可以使用 3 个 worker 和 3 个线程来得到最大为 9 的并发请求数量。

$ gunicorn --workers=3 --threads=3 main:app

第三种并发方式（“伪线程”）

一些 Python 库比如（gevent 和 Asyncio）可以在 Python 中启用多并发。那是基于协程实现的“伪线程”。Gunicrn 允许通过设置对应的 worker 类来使用这些异步 Python 库。

这里的设置适用于我们想要在单核机器上运行的gevent：

$ gunicorn --worker-class=gevent --worker-connections=1000 --workers=3 main:app
  • worker-connection 是对于 gevent worker 类的特殊设置，每个workers的最大连接数。
  • --workers 工作线程数，建议为（2CPU）+1。因为这里是单核，我们设置的是3个worker。

错误

启动后访问接口，报：TypeError: FastAPI.__call__() missing 1 required positional argument: 'send'的错误

如果启动后访问接口，报错：

TypeError: FastAPI.__call__() missing 1 required positional argument: 'send'
[2024-04-08 16:20:33 +0800] [21554] [ERROR] Error handling request /favicon.ico

则可以确认一下，启动时是否指定：-k uvicorn.workers.UvicornWorker

如果启动时报：gunicorn: error: unrecognized arguments: -w 4 - k uvicorn.workers.UvicornWorker main: server等类似错误。

可以搜索一下代码里面是否使用了parser.parse_args()获取参数，这种获取参数的方式，会将gunicorn的启动参数当做输入参数。

使用parser.parse_args()这种获取参数的方式时，在使用gunicorn启动时会报上面的错误。

# 解析命令行参数
args = parser.parse_args()
# 设置环境变量
# uvicorn模式启动，读取的.env*里面的APP_ENV
os.environ["APP_ENV"] = args.env

使用args, unknow = parser.parse_known_args()替换之后就可以了。

# 解析命令行参数
args, unknow = parser.parse_known_args()
# 设置环境变量
# uvicorn模式启动，读取的.env*里面的APP_ENV
os.environ["APP_ENV"] = args.env
